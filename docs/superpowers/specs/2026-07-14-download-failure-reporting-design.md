# 下载阶段结果同步输出设计

## 目标

`ac.sh install` 或 `ac.sh update` 的下载阶段结束时，将最终结果同时输出到控制台和主日志 `ac.log`，并继续在 `downloads.log` 中保留全部候选源的尝试记录。

## 输出行为

每个下载对象成功后立即输出一行，列出对象名称、实际使用的下载地址和目标：

```text
[OK] 下载: AzerothCore 源码 完成: <实际下载地址> -> <目标文件>
[OK] 下载: mod-playerbots 源码 完成: <实际下载地址> -> <目标文件>
[OK] 下载: 客户端数据 完成: <实际下载地址> -> <目标文件>
[OK] 下载: Docker 镜像 完成: <实际镜像地址> -> <目标镜像>
```

使用本地文件或镜像缓存时，将实际来源标记为“本地缓存”：

```text
[OK] 下载: AzerothCore 源码 完成: 本地缓存 -> <目标文件>
```

输出项按实际配置动态生成，不固定模块或镜像数量。

全部候选源均失败时，向标准错误输出：

```text
[ERROR] 下载失败: <目标文件或镜像>
最后下载源: <URL 或镜像地址>
具体错误:
<curl 或 Docker 返回的原始错误>
```

`ac.sh` 已将标准输出和标准错误通过 `tee` 同时写入控制台与 `ac.log`，因此下载层只负责准确生成最终错误，不重复实现主日志写入。

## 实现边界

- 文件下载保存最后一个候选 URL 及其 `curl` 错误；ZIP 校验失败时生成明确的校验错误。
- 客户端最新版本解析保存最后一个候选 URL，以及请求失败或响应头无法解析的具体原因。
- Docker 镜像拉取保存最后一个候选镜像地址及 `docker pull` 错误。
- 每个成功下载或缓存复用项立即按统一格式输出到控制台和 `ac.log`。
- 下载阶段的标准错误不再仅被 `download_log_output` 消费，确保最终错误能继续传递给 `ac.sh`。
- `downloads.log` 继续记录所有候选源、每次失败详情和最终摘要。

## 测试

使用 Bash 测试和命令替身模拟下载失败，至少验证：

- 多个候选源均失败时，只在最终结果中报告最后一个源的原始错误。
- 控制台失败输出包含目标、最后下载源和具体错误。
- `downloads.log` 包含所有候选源的失败记录。
- 客户端版本解析失败和 Docker 镜像拉取失败遵循同一输出约定。
- 每个对象成功后立即输出一次，并准确列出对象名称、实际来源和目标。
- 使用本地缓存时，成功结果将来源标记为“本地缓存”。
- 全部 Shell 文件通过 `bash -n`，已有成功路径不受影响。

## 非目标

- 不把下载过程实时输出到控制台。
- 不改变镜像选择、重试顺序、超时参数或缓存策略。
- 不修改一键安装入口 `install.sh` 的源码归档下载行为。
- 不在最终结果中输出 `downloads.log` 路径。
